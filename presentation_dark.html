Purpose

This catalog defines the user scenarios for the JFT Skills Development Platform (JFT SDP) MVP release. It is scoped to the four MVP scenarios that constitute the complete functional scope of the first production deployment.

The MVP objective is to deploy the existing v1 proof-of-concept codebase (Cicada) to a production-quality, scalable infrastructure with a polished UI, accessible outside the WGU intranet via LTI 1.3 single sign-on. No new features, adaptive logic changes, or coaching algorithm modifications are in scope for this release.

Each of the four scenarios is self-contained. Each represents a distinct student profile interacting with the Coding Coach for the first time (SC-MVP-01, 02, 03) or returning after a break (SC-MVP-04). Every scenario begins from the LMS course page and includes its own complete LTI launch, diagnostic or re-assessment, and coaching flow.

13 Learning Objective Sub-Sections

The coaching structure for this release is defined by these 13 sub-sections, inherited from the v1 Cicada codebase:

1. Basic Syntax and Data Types
2. Control Flow and Logic
3. Functions and Modular Programming
4. Data Structures: Lists, Tuples, Sets, Dicts
5. Working with Files and I/O
6. Error Handling and Debugging
7. Object-Oriented Programming (OOP)
8. Python Libraries and Modules
9. Advanced Data Handling
10. Regular Expressions and Strings
11. Working with Databases
12. Introduction to Automation
13. Introduction to Machine Learning
SC-MVP-01

First Launch of the Coding Coach

Sally (Basic) — No prior Python knowledge · 8 Screens
Scenario IDSC-MVP-01
User TypeStudent
PersonaSally (Basic — no prior Python knowledge)
Storyboard Screens8 (Screen IDs: 1, 2, 3, 4, 5, 6, 7, 8)

Context

Sally is an enrolled WGU student working through E010: Foundations of Programming (Python). The JFT SDP coding coach is available to her as a supplemental practice tool via a resource link in the course she can click at any time. It does not replace any formative opportunity or high-stakes course assessment. A stable LTI 1.3 resource link has been placed in E010 by the D&D team using LRPS. Sally has no prior Python knowledge. She clicks the Coding Coach link for the first time.

Trigger

Sally clicks the "Coding Coach — Python" resource link in E010. The LTI 1.3 link launch completes, WGU SSO confirms her identity silently (no separate login prompt), and the coaching platform opens in a new browser tab.

A “Coding Coach” button is positioned next to the Competencies section. Sally clicks the Coding Coach button.

Session Saved
Screen 8

Sally closes the tool. The confirmation card shows: sub-section (Basic Syntax & Data Types), tasks completed (1), sub-sections unlocked (1 of 13), session duration (14 min), resume point (Variables & Assignment — Task 2). “Return to E010 Course” and “View Progress Map” links are available. A footer note reads: “This tool does not affect your course grade or academic record.” If Sally exits mid-task rather than at a natural task boundary, the platform resumes her at the last completed task boundary on her next launch — not at the mid-task exit point. Mid-task state within an incomplete question is not preserved.

Expected Outcome

  • LTI 1.3 basic launch completes without a separate login prompt.
  • The diagnostic samples across all 13 SDP sub-sections.
  • A coaching baseline is stored against Sally's WGU student ID via LTI 1.3.
  • Sally is routed to an appropriate first coaching task based on diagnostic results.
  • Session progress is saved on exit. On re-launch, Sally resumes at the last completed task boundary.
  • All screens meet WCAG 2.2 AA and are navigable by keyboard and screen reader.

Notes for JFT

  • The coding coach does not generate grades, feed grade passback to the LMS, or appear in any student academic record. WGU is using basic LTI 1.3. There is no LTI Advantage.
  • The D&D team places and manages resource links in the LMS using WGU’s Learning Resource Provisioning System (LRPS). LRPS can generate authenticated LTI-standard links to any vendor-owned resource, or act as a passthrough for any public URL. When a student clicks an LRPS link from a course content page, the system handles SSO transparently (students are typically already signed in) and redirects to the target resource. The LTI 1.3 technical integration between LRPS and the JFT SDP platform is handled by WGU’s tools integration team. JFT does not own LMS-side link placement. The platform must expose a stable launch URL that does not change when a course’s content or module structure is reorganized.
  • Diagnostic question generation relies on LLM-inherent variation to prevent identical question sets across students. JFT must document the generation strategy in the technical design. Additional signals — such as time-to-response and first-attempt accuracy — may be used by the existing Cicada v1 logic to inform the competency model; JFT must document how the v1 codebase handles these signals.
  • FERPA requirements apply to all stored student interaction data from the moment of first launch. Minimum data stored in the MVP: WGU student ID (from LTI), sub-section progress indicators, and session timestamps. The MVP stores competency-level progress data (e.g., which sub-sections a student has passed and at what difficulty level), not individual interaction state (e.g., the full text of LLM-generated questions or student responses). JFT must document the data storage approach and confirm FERPA compliance before production deployment. The LRPS integration and its role in the LTI launch flow were reviewed with JFT by the WGU LRPS team on 06 Apr 2026.
  • The storyboard shows the zyBooks course page (Screen 1) as a design reference for contextualizing the LTI launch. This screen is not part of the Coding Coach application — it illustrates the student’s starting point in the zyBooks textbook.
SC-MVP-02

Progressive Coaching with Targeted Feedback

Sally (Advanced) — Partial Python knowledge · 11 Screens
Scenario IDSC-MVP-02
User TypeStudent
PersonaSally (Advanced — partial Python knowledge)

Context

Sally is an enrolled WGU student working through E010: Foundations of Programming (Python). She has partial Python knowledge — she understands basic syntax but has gaps in data structures and more advanced topics. She accesses the Coding Coach for the first time. This scenario demonstrates progressive coaching with targeted feedback when a student answers incorrectly, including gap verification and difficulty advancement.

Screen 16

A new question at the same difficulty verifies gap resolution. Sally is asked to filter the student_scores dictionary, printing only students who scored 90 or above.

Verification Passed — Difficulty Advances
Screen 18

Sally answers correctly. "Verification Passed — Gap resolved — difficulty advancing." Progression: Foundational → Intermediate → Advanced. Identical questions are never repeated within a session. Throughout coaching, a persistent session-stats display shows the three canonical fields — Sub-section (e.g., "Data Structures"), Tasks completed (e.g., 3), and Current difficulty (e.g., "Intermediate") — so Sally always knows her progression state without leaving the active task.

11
Session Saved
Screen 19

Confirmation: Data Structures, 4 tasks completed, Intermediate difficulty, 2 of 13 sub-sections explored, 22 min session, resume at Dictionary Comprehensions — Task 6.

Expected Outcome

  • Difficulty increases progressively and does not repeat identical questions within a session.
  • Incorrect-answer feedback is specific to Sally's actual error, not generic.
  • Session progress is fully preserved. Sally resumes at her exact competency position.
  • Progress map updates in real time.
  • Verification questions confirm gap resolution before difficulty advances.

Notes for JFT

  • Feedback specificity is the primary quality risk for the MVP coaching experience. JFT must demonstrate (if it is not present in the PoC codebase) in future iterations that feedback varies meaningfully across different incorrect answers to the same question. A templated response that ignores what the student actually wrote does not satisfy this requirement.
  • The difficulty ceiling for each SDP sub-section is defined by the existing v1 Cicada codebase and is inherited as-is for the MVP. JFT must document how difficulty levels are parameterized in the LLM prompt and flag any sub-sections where the ceiling logic requires calibration before production deployment. We will use this documentation to determine future iterations/capabilities, such as the need to re-build the difficulty logic for other programming languages/subjects.
  • Session progress persistence strategy must be documented in JFT’s technical design. FERPA requirements apply to all stored student interaction data. The MVP does not store exact question-and-response state; it stores competency-level progress indicators and session timestamps against the student’s WGU ID.
  • The storyboard shows three diagnostic questions to illustrate the range of difficulty and sub-section sampling. The actual diagnostic may contain up to 10 questions.

Expected Outcome

  • Sally is not routed into coaching content for sub-sections where mastery was demonstrated.
  • At least one sub-section requires verification beyond the diagnostic.
  • The full fast-track path is achievable in one short session.
  • From the no-gaps summary, Sally can browse sub-sections for targeted practice.

Notes for JFT

  • Fast-track threshold logic is inherited from the v1 Cicada codebase as-is. JFT must document the existing threshold parameters — specifically what diagnostic performance level triggers automatic fast-track versus routing into standard coaching — in the technical design document before UAT.
  • The MVP does not include an in-browser code execution environment. All submitted code is evaluated by the LLM as text (this is an assumption based on UI interaction of v1 code). JFT should flag any sub-sections in the v1 codebase where text-based LLM evaluation of code is insufficient for reliable competency assessment.
  • Student-directed sub-section selection — allowing a student to choose which sub-section to practice regardless of platform routing — is supported in the MVP.
  • Fast-tracked students retain access to the coaching tool throughout their course enrollment. The tool is available for continued practice from the student’s current competency state at any time.
SC-MVP-04

Student Returns and Resumes a Prior Session

Sally (Returning) — Resuming after multi-week break · 6 Screens
Scenario IDSC-MVP-04
User TypeStudent
PersonaSally (Returning — resuming after multi-week break)
Storyboard Screens6 (Screen IDs: 29, 30, 31, 32, 33, 34)

Context

Sally worked through two coaching sessions in her first week of E010, making intermediate progress in two SDP sub-sections (Basic Syntax & Data Types completed baseline; Data Structures: Lists, Tuples, Sets, and Dictionaries in progress at intermediate level). She then paused for several weeks due to work commitments. She is now returning to the coding coach. Session progress is preserved and Sally resumes at the last completed task boundary. A brief re-assessment verifies retention before coaching resumes at her prior difficulty level.

Trigger

Resume Coaching Task
Screen 33

Sally picks up Data Structures at Intermediate. Breadcrumb: "Coding Coach > Data Structures > Dictionary Operations." Prompt: filter a dictionary by computed criteria — return students above the class average. Tip: use sum() and len() for the average.

6
Session Saved
Screen 34

Data Structures, 2 tasks, Intermediate, 2 of 13 explored, 18 min, resume at Dictionary Operations — Task 8.

Expected Outcome

  • Prior progress is preserved and surfaced immediately. No restart from initial diagnostic.
  • Returning-learner screen shows last session date, active sub-section, difficulty, and resume point.
  • Brief re-assessment (2 questions) verifies retention before coaching resumes.
  • Coaching resumes at last completed task boundary at prior difficulty level.
  • If performance indicates decay, difficulty adjusts downward per Cicada v1 logic silently.

Notes for JFT

  • Session progress persistence is a foundational MVP requirement. If session progress is lost on re-entry — for any reason, including infrastructure changes or deployment updates — the tool fails its primary value proposition for returning students.
  • Configurable re-assessment thresholds after extended breaks are a future capability (see JFT SDP User Scenario Catalog v1.1, SC-FUT-01). The brief re-assessment shown in the storyboard verifies retention at the student’s prior difficulty level; it does not recalibrate the full diagnostic baseline.
  • WGU’s rolling enrollment means students have highly variable break patterns. Session persistence must handle gaps of any length without data loss. FERPA data retention requirements mandate that coaching session data be retained for up to three years from the session date. A student returning after more than three years has no retained data and is treated as a new user.
  • In-session regression behavior is handled by the adaptive difficulty algorithm in the v1 Cicada codebase: the algorithm reduces question difficulty in response to sustained incorrect answers. No explicit regression detection logic or student notification is added in the MVP. JFT must document how the v1 difficulty progression algorithm handles sustained regression patterns so that this behavior is testable during UAT.

Six post-MVP scenarios · 39 new screens across 3 per-persona portals

Source
