MVP Scenario Catalog v1.2 · 01 Apr 2026 · SDP Design System v1.2 · Storyboard v3
Author: Brady Redfearn, Senior Strategist, Program Development

Purpose

This catalog defines the user scenarios for the JFT Skills Development Platform (JFT SDP) MVP release. It is scoped to the four MVP scenarios that constitute the complete functional scope of the first production deployment.

The MVP objective is to deploy the existing v1 proof-of-concept codebase (Cicada) to a production-quality, scalable infrastructure with a polished UI, accessible outside the WGU intranet via LTI 1.3 single sign-on. No new features, adaptive logic changes, or coaching algorithm modifications are in scope for this release.

Each of the four scenarios is self-contained. Each represents a distinct student profile interacting with the Coding Coach for the first time (SC-MVP-01, 02, 03) or returning after a break (SC-MVP-04). Every scenario begins from the LMS course page and includes its own complete LTI launch, diagnostic or re-assessment, and coaching flow.

13 Learning Objective Sub-Sections

The coaching structure for this release is defined by these 13 sub-sections, inherited from the v1 Cicada codebase:

1. Basic Syntax and Data Types
2. Control Flow and Logic
3. Functions and Modular Programming
4. Data Structures: Lists, Tuples, Sets, Dicts
5. Working with Files and I/O
6. Error Handling and Debugging
7. Object-Oriented Programming (OOP)
8. Python Libraries and Modules
9. Advanced Data Handling
10. Regular Expressions and Strings
11. Working with Databases
12. Introduction to Automation
13. Introduction to Machine Learning
SC-MVP-01

First Launch of the Coding Coach

Sally (Basic) — No prior Python knowledge · 8 Screens
Scenario IDSC-MVP-01
User TypeStudent
PersonaSally (Basic — no prior Python knowledge)
Storyboard Screens8 (Screen IDs: 1, 2, 3, 18, 4, 5, 6, 7)

Context

Sally is an enrolled WGU student working through E010: Foundations of Programming (Python). The JFT SDP coding coach is available to her as a supplemental practice tool via a resource link in the course she can click at any time. It does not replace any formative opportunity or high-stakes course assessment. Sally has no prior Python knowledge. She clicks the Coding Coach link for the first time.

Trigger

Sally clicks the "Coding Coach — Python" resource link in E010. The LTI 1.3 link launch completes, WGU SSO confirms her identity silently (no separate login prompt), and the coaching platform opens in a new browser tab.

1
LMS Course Page
Screen 1

Sally is already signed into her WGU course (E010: Foundations of Programming — Python). The LMS displays a mock textbook page for Module 1: Python Basics with course navigation in a left sidebar. The sidebar includes course modules (Python Basics, Control Flow, Functions, Data Structures), supplemental resources (Course Textbook, Video Lectures), and a highlighted "Coding Coach — Python" link under Practice Tools. A brief description reads: "AI-powered Python practice. Launches via LTI in a new tab. No separate login required." Sally clicks the Coding Coach link.

2
Welcome / Entry Screen
Screen 2

The Coding Coach opens in a new browser tab. The entry screen displays a badge ("Supplemental Practice Tool"), the heading "Coding Coach — Python Foundations", and a plain-language description. A "Begin Diagnostic" button is the primary CTA with a note "(Takes about 5-8 minutes)." Trust signals appear below: "Secured via WGU SSO," "WCAG 2.2 AA," "Works on any device." The right side lists all 13 practice areas. No tutorial is required.

3
Diagnostic Question 1
Screen 3

The platform presents a short diagnostic — LLM-generated questions sampling across all 13 sub-sections. The first screen shows Question 3 of 10, sub-section "Basic Syntax & Data Types," with a progress bar at 30%. The AI-generated prompt asks Sally to define and use variables in Python. A dark-themed code editor provides space for her response. A "Need a Hint?" link and "Submit Answer" button are available.

4
Diagnostic Question 2
Screen 18

Question 7 of 10, sub-section "Control Flow & Logic" at 70%. The prompt asks Sally to write a Python function called check_grade that takes a numeric score and returns "Pass" or "Fail." Questions are calibrated for a student with zero prior Python experience.

5
Diagnostic Results
Screen 4

Sally completes the diagnostic. The results screen displays "We've established your starting point." Her results show no measurable prior knowledge across any sub-section. The baseline is stored against Sally's WGU student ID via LTI 1.3. All 13 sub-sections are listed as chips. A "View Your Progress Map" CTA leads to the next screen.

6
Progress Map
Screen 5

The progress map shows "13 sub-sections · 0 mastered · 13 to explore." All show "Not Started." Basic Syntax & Data Types is highlighted as the "Recommended starting point." Sally can "Start Coaching" or "Return to Course Textbook."

7
First Coaching Task
Screen 6

Sally is routed to "Variables & Assignment" in Basic Syntax & Data Types. The AI coaching prompt introduces variables and asks Sally to create a student_name variable. A tip box provides guidance. The response area includes a code editor, "Submit Answer," "Need a Hint?" link, and "Save & Exit" ghost button.

8
Session Saved
Screen 7

Sally closes the tool. The confirmation card shows: sub-section (Basic Syntax & Data Types), tasks completed (1), sub-sections unlocked (1 of 13), session duration (14 min), resume point (Variables & Assignment — Task 2). "Return to E010 Course" and "View Progress Map" links are available. If Sally exits mid-task, she resumes at the last completed task boundary on her next launch.

Expected Outcome

  • LTI 1.3 basic launch completes without a separate login prompt.
  • The diagnostic samples across all 13 SDP sub-sections.
  • A coaching baseline is stored against Sally's WGU student ID via LTI 1.3.
  • Sally is routed to an appropriate first coaching task based on diagnostic results.
  • Session progress is saved on exit. On re-launch, Sally resumes at the last completed task boundary.
  • All screens meet WCAG 2.2 AA and are navigable by keyboard and screen reader.

Notes for JFT

  • The coding coach does not generate grades or feed passback to the LMS. WGU is using basic LTI 1.3 (no LTI Advantage).
  • The platform must expose a stable launch URL that does not change when course structure is reorganized.
  • Diagnostic question generation relies on LLM-inherent variation to prevent identical question sets.
  • FERPA requirements apply to all stored student interaction data from first launch.
SC-MVP-02

Progressive Coaching with Targeted Feedback

Sally (Advanced) — Partial Python knowledge · 11 Screens
Scenario IDSC-MVP-02
User TypeStudent
PersonaSally (Advanced — partial Python knowledge)
Storyboard Screens11 (Screen IDs: 15, 25, 26, 27, 31, 8, 19, 9, 10, 20, 21)

Context

Sally has partial Python knowledge — she understands basic syntax but has gaps in data structures and more advanced topics. She accesses the Coding Coach for the first time. This scenario demonstrates progressive coaching with targeted feedback when a student answers incorrectly, including gap verification and difficulty advancement.

Trigger

Sally clicks the "Coding Coach — Python" resource link in E010. The LTI 1.3 launch completes, WGU SSO confirms her identity silently, and the coaching platform opens in a new browser tab.

1
LMS Course Page
Screen 15

Sally is signed into E010. The LMS displays the mock textbook page with the "Coding Coach — Python" link. Sally clicks the link.

2
Welcome / Entry Screen
Screen 25

The Coding Coach opens with the entry screen: "Begin Diagnostic" CTA, trust signals, and 13 practice area list. Sally clicks "Begin Diagnostic."

3
Diagnostic Question 1
Screen 26

Question 3 of 10, "Basic Syntax & Data Types" at 30%. Prompt asks Sally to define and use variables.

4
Diagnostic Question 2
Screen 27

Question 7 of 10, "Control Flow & Logic" at 70%. Prompt asks Sally to write a check_grade function.

5
Diagnostic Question 3
Screen 31

"Data Structures: Lists, Tuples, Sets, Dictionaries" — a more advanced diagnostic question reflecting Sally's partial knowledge.

6
Progress Map
Screen 8

Sally's diagnostic reveals partial knowledge. The progress map shows: "13 sub-sections · 1 in progress · 12 not started." Data Structures is marked "In Progress — Foundational level." Basic Syntax shows "Completed diagnostic baseline."

7
Correct Answer at Foundational Level
Screen 19

The AI prompt asks for the Python syntax to append 'grape' to a list. Sally answers correctly with fruits.append('grape'). The platform confirms with specific feedback and notes difficulty will increase.

8
Incorrect Answer + Targeted Feedback
Screen 9

At increased difficulty on dictionary iteration, Sally answers incorrectly — her code prints the entire dictionary instead of individual key-value pairs. The platform generates targeted feedback identifying the specific error and provides a concrete correct example using .items() and f-string formatting. "Try Next Question" at same difficulty level.

9
Verification Question (Same Difficulty)
Screen 10

A new question at the same difficulty verifies gap resolution. Sally is asked to filter the student_scores dictionary, printing only students who scored 90 or above.

10
Verification Passed — Difficulty Advances
Screen 20

Sally answers correctly. "Verification Passed — Gap resolved — difficulty advancing." Progression: Foundational → Intermediate → Advanced. Identical questions are never repeated within a session.

11
Session Saved
Screen 21

Confirmation: Data Structures, 4 tasks completed, Intermediate difficulty, 2 of 13 sub-sections explored, 22 min session, resume at Dictionary Comprehensions — Task 6.

Expected Outcome

  • Difficulty increases progressively and does not repeat identical questions within a session.
  • Incorrect-answer feedback is specific to Sally's actual error, not generic.
  • Session progress is fully preserved. Sally resumes at her exact competency position.
  • Progress map updates in real time.
  • Verification questions confirm gap resolution before difficulty advances.

Notes for JFT

  • Feedback specificity is the primary quality risk. Templated responses do not satisfy this requirement.
  • Difficulty ceiling per sub-section is inherited from Cicada v1 as-is.
  • FERPA requirements apply. MVP stores competency-level indicators, not exact Q&A state.
SC-MVP-03

Experienced Developer Fast-Tracks Through Coaching Baseline

Sally (Professional) — Experienced Python developer · 9 Screens
Scenario IDSC-MVP-03
User TypeStudent
PersonaSally (Professional — experienced Python developer)
Storyboard Screens9 (Screen IDs: 16, 28, 29, 30, 33, 22, 11, 12, 13)

Context

Sally is a software developer enrolled in E010 with several years of professional Python experience. She accesses the coding coach expecting it to recognize her existing skills quickly so she is not required to work through content she does not need.

Trigger

Sally clicks the "Coding Coach — Python" resource link in E010. The LTI 1.3 launch completes and the diagnostic opens.

1
LMS Course Page
Screen 16

Sally is signed into E010. She clicks the "Coding Coach — Python" link.

2
Welcome / Entry Screen
Screen 28

The entry screen with "Begin Diagnostic" CTA. Sally clicks to start.

3
Diagnostic Question 1
Screen 29

Question 3 of 10, "Basic Syntax & Data Types." Sally's professional experience means she completes this quickly.

4
Diagnostic Question 2
Screen 30

Question 7 of 10, "Control Flow & Logic." Prompt asks to write a check_grade function.

5
Diagnostic Question 3
Screen 33

"Data Structures: Lists, Tuples, Sets, Dictionaries" — advanced diagnostic question.

6
Advanced Diagnostic with Positive Feedback
Screen 22

Question 8 of 10, "Advanced Data Handling" at 80%. Sally writes a list comprehension with compound filtering. The platform responds: "Your response demonstrates fluent use of list comprehensions. This is characteristic of an experienced Python developer. The diagnostic is calibrating your baseline."

7
Fast-Track Diagnostic Results
Screen 11

"Strong baseline detected." 12 of 13 sub-sections show mastery. Functions & Modular Programming requires hands-on verification. "Begin Verification Task" CTA.

8
Verification Task (Write a Python Module)
Screen 12

Sally must create a math_utils.py module with add and multiply functions, a __name__ == "__main__" guard, and demonstrate importing. Alert: "Your code is evaluated by the AI as text. There is no in-browser code execution environment."

9
No-Gaps Summary
Screen 13

"No coaching gaps identified. Proficiency confirmed across all 13 sub-sections." Options: continue practicing, select any sub-section for targeted practice, or return to E010.

Expected Outcome

  • Sally is not routed into coaching content for sub-sections where mastery was demonstrated.
  • At least one sub-section requires verification beyond the diagnostic.
  • The full fast-track path is achievable in one short session.
  • From the no-gaps summary, Sally can browse sub-sections for targeted practice.

Notes for JFT

  • Fast-track threshold logic is inherited from Cicada v1. JFT must document the threshold parameters.
  • The MVP does not include in-browser code execution. All code is evaluated by the LLM as text.
  • Student-directed sub-section selection is supported in the MVP.
SC-MVP-04

Student Returns and Resumes a Prior Session

Sally (Returning) — Resuming after multi-week break · 6 Screens
Scenario IDSC-MVP-04
User TypeStudent
PersonaSally (Returning — resuming after multi-week break)
Storyboard Screens6 (Screen IDs: 17, 14, 34, 15*, 23, 24)

Context

Sally worked through two coaching sessions making intermediate progress in two sub-sections. She paused for several weeks. Session progress is preserved and she resumes at the last completed task boundary. A brief re-assessment verifies retention before coaching resumes at her prior difficulty level.

Trigger

Sally clicks the "Coding Coach — Python" resource link in E010 after a multi-week break. The LTI 1.3 launch completes.

1
LMS Course Page
Screen 17

Sally is signed into E010. She clicks the "Coding Coach — Python" link.

2
Returning Learner Welcome
Screen 14

"Welcome back, Sally." Session snapshot: Last session (3 weeks ago), Active sub-section (Data Structures at Intermediate), Resume point (Dictionary Iteration — Task 5), Sub-sections explored (2 of 13). Full progress map with all 13 sub-sections and current status. "Resume Coaching" and "Return to Course" buttons available.

3
Re-Assessment Question 1
Screen 34

Brief re-assessment to verify retention. Question 1 of 2, "Data Structures" with badge "Re-Assessment · Welcome Back." Prompt: access and transform nested dictionary data — print each student's name and average grade. Topic: "Data Structures — Nested Dictionaries (Intermediate)."

4
Re-Assessment Question 2
Screen 15*

Question 2 of 2, "Control Flow & Logic" at 100%. Write a function count_passing that returns pass/fail counts from a list of scores. Topic: "Control Flow & Logic — Loops and Conditionals (Intermediate)."

5
Resume Coaching Task
Screen 23

Sally picks up Data Structures at Intermediate. Breadcrumb: "Coding Coach > Data Structures > Dictionary Operations." Prompt: filter a dictionary by computed criteria — return students above the class average. Tip: use sum() and len() for the average.

6
Session Saved
Screen 24

Data Structures, 2 tasks, Intermediate, 2 of 13 explored, 18 min, resume at Dictionary Operations — Task 8.

Expected Outcome

  • Prior progress is preserved and surfaced immediately. No restart from initial diagnostic.
  • Returning-learner screen shows last session date, active sub-section, difficulty, and resume point.
  • Brief re-assessment (2 questions) verifies retention before coaching resumes.
  • Coaching resumes at last completed task boundary at prior difficulty level.
  • If performance indicates decay, difficulty adjusts downward per Cicada v1 logic silently.

Notes for JFT

  • Session persistence is a foundational MVP requirement. Lost progress = tool failure.
  • Configurable re-assessment thresholds are a future capability. The brief re-assessment shown verifies retention only.
  • Session persistence must handle gaps of any length. FERPA requires 3-year data retention.
  • JFT must document how the v1 difficulty algorithm handles sustained regression patterns.
